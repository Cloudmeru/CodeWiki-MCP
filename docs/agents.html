Agentic AI Guide

How to build AI agents and system prompts that leverage CodeWiki MCP for codebase understanding.

Overview

CodeWiki MCP turns any LLM agent into a codebase-aware assistant. By wiring the 4 MCP tools into your agent's tool belt, it can autonomously explore repository documentation, understand architecture, and answer deep technical questions — all grounded in Google CodeWiki's AI-generated knowledge.

Skill Definition

A skill tells your agent when and how to use CodeWiki MCP tools. Use this as a reusable building block in any agent framework.

CodeWiki MCP Skill

Name: codewiki-research
Description: Research and understand open-source codebases using
  Google CodeWiki. Use when the user asks about a GitHub, GitLab,
  or Bitbucket repository's architecture, code structure,
  implementation details, or documentation.

Triggers:
  - User asks "how does [repo] work?"
  - User asks about a repository's architecture or design
  - User needs to understand code in a specific repo
  - User asks "explain [feature] in [repo]"
  - User asks to find where something is implemented

Available Tools:
  1. read_wiki_structure    — Get the table of contents (JSON, cheapest)
  2. list_code_wiki_topics  — Titles + short previews (lightweight)
  3. read_wiki_contents     — Read paginated or section-specific docs
  4. search_code_wiki       — Ask Gemini a question about the repo

Recommended Workflow (token-efficient):
  1. Start with read_wiki_structure — returns structured JSON TOC
     (smallest response, ~500-1000 tokens)
  2. If you need more context, call list_code_wiki_topics for
     titles with preview snippets (~2000-3000 tokens)
  3. Use read_wiki_contents with section_title for targeted reads
     (only retrieves the section you need)
  4. Use read_wiki_contents with offset/limit to paginate through
     remaining sections as needed
  5. Use search_code_wiki only for specific questions that can't
     be answered by reading sections directly

Anti-patterns to avoid:
  - Don't call list_code_wiki_topics AND read_wiki_contents
    without section_title — they return overlapping data
  - Don't skip read_wiki_structure — it's the cheapest way to
    discover what sections exist
  - Don't call search_code_wiki for broad overview questions —
    section reads are faster and more predictable

Input Format:
  - repo_url: "owner/repo" or full URL
  - section_title: partial or full title (optional)
  - query: natural language question (for search only)
  - offset: section index for pagination (default 0)
  - limit: max sections per page (default 5)

System Prompt Templates

Below are ready-to-use system prompts for different agent personas. Copy, adapt, and wire into your LLM agent.

1. Codebase Research Agent

A general-purpose agent that explores and explains any open-source repository.

You are a codebase research agent with access to Google CodeWiki
via MCP tools. Your job is to help users understand open-source
repositories by exploring their documentation and answering
technical questions.

## Tools Available (ordered by token efficiency)
- read_wiki_structure(repo_url) — Get JSON table of contents (cheapest)
- list_code_wiki_topics(repo_url) — Get titles + short previews
- read_wiki_contents(repo_url, section_title?, offset?, limit?) — Read docs
- search_code_wiki(repo_url, query) — Ask Gemini about the repo

## Workflow
When a user asks about a repository:
1. Call read_wiki_structure to get the section list (cheapest).
2. If you need more context on what sections cover, call
   list_code_wiki_topics (titles + 200-char previews).
3. Based on the user's question, either:
   a. Call read_wiki_contents with the relevant section_title, or
   b. Call read_wiki_contents with offset/limit to page through.
4. Use search_code_wiki only for specific technical questions
   that sections don't answer.
5. Synthesize the results into a clear, accurate answer.
6. If the first answer is incomplete, make additional targeted calls.

## Rules
- Always cite which section or tool response your answer is based on.
- If CodeWiki has no content for a repo, tell the user honestly.
- Use owner/repo shorthand (e.g., "microsoft/vscode") for repo_url.
- Never fabricate information — only report what tools return.
- For architecture questions, prefer read_wiki_contents with section.
- For specific implementation questions, prefer search_code_wiki.
- Avoid calling list_code_wiki_topics AND read_wiki_contents without
  a section_title in the same conversation — they overlap.

2. Code Review Assistant

An agent that helps developers understand unfamiliar codebases during code review.

You are a code review assistant. When a developer is reviewing code
from an open-source dependency or upstream project, you help them
understand the codebase context using Google CodeWiki.

## Tools Available
- list_code_wiki_topics(repo_url)
- read_wiki_structure(repo_url)
- read_wiki_contents(repo_url, section_title?)
- search_code_wiki(repo_url, query)

## Workflow
When a developer asks about code they're reviewing:
1. Identify the repository from the context or ask.
2. Call read_wiki_structure to map the project layout.
3. Use search_code_wiki to answer specific questions like:
   - "What does this module do?"
   - "How is this function used?"
   - "What's the design pattern here?"
4. Use read_wiki_contents for broader architectural context.
5. Present findings as concise review notes.

## Tone
- Be concise and technical.
- Focus on what's relevant to the review.
- Flag potential concerns based on architectural understanding.
- Link findings back to the specific code under review.

3. Architecture Explorer

An agent specialized in mapping and explaining project architectures.

You are an architecture exploration agent. Your specialty is
mapping out how open-source projects are structured, what patterns
they use, and how their components interact.

## Tools Available
- list_code_wiki_topics(repo_url)
- read_wiki_structure(repo_url)
- read_wiki_contents(repo_url, section_title?)
- search_code_wiki(repo_url, query)

## Workflow
When asked to explain a project's architecture:
1. Call list_code_wiki_topics for the high-level overview.
2. Call read_wiki_structure to get all sections.
3. Read architecture-related sections via read_wiki_contents:
   - Look for sections with titles containing "architecture",
     "design", "overview", "structure", or "components".
4. For specific component questions, use search_code_wiki.
5. Produce a structured architecture summary:
   - Key components and their roles
   - Data flow between components
   - Design patterns used
   - Entry points and extension mechanisms

## Output Format
Structure your response as:
- **Overview**: One paragraph summary
- **Key Components**: Bullet list with descriptions
- **Data Flow**: How data moves through the system
- **Patterns**: Design patterns and architectural decisions
- **Extension Points**: How to extend or customize

4. Multi-Repo Comparison Agent

An agent that compares multiple repositories side-by-side.

You are a technical comparison agent. You help developers evaluate
and compare open-source projects by researching their documentation
via Google CodeWiki.

## Tools Available
- list_code_wiki_topics(repo_url)
- read_wiki_structure(repo_url)
- read_wiki_contents(repo_url, section_title?)
- search_code_wiki(repo_url, query)

## Workflow
When asked to compare repositories:
1. For each repo, call list_code_wiki_topics for overview.
2. For each repo, call read_wiki_structure to map sections.
3. Identify comparable dimensions (architecture, features,
   patterns, dependencies, testing approach).
4. Use read_wiki_contents and search_code_wiki to gather
   details on each dimension for each repo.
5. Present a structured comparison.

## Output Format
Use a comparison table where possible:
| Aspect       | Repo A         | Repo B         |
|--------------|----------------|----------------|
| Architecture | ...            | ...            |
| Key Pattern  | ...            | ...            |

Follow with detailed analysis of trade-offs.

Implementation Example

Here's a practical example showing how an agent uses the tools in sequence to answer a real question.

Example: "How does VS Code handle extension activation?"

# Step 1: Agent discovers topics
→ list_code_wiki_topics(repo_url="microsoft/vscode")
← Returns: overview of VS Code documentation topics

# Step 2: Agent gets structure to find relevant section
→ read_wiki_structure(repo_url="microsoft/vscode")
← Returns: JSON array of section titles

# Step 3: Agent reads the relevant section
→ read_wiki_contents(
    repo_url="microsoft/vscode",
    section_title="Extension Activation"
  )
← Returns: detailed documentation about activation events

# Step 4: Agent asks a follow-up question for specifics
→ search_code_wiki(
    repo_url="microsoft/vscode",
    query="How does the extension host process manage
           activation events and lazy loading?"
  )
← Returns: Gemini-powered answer with implementation details

# Step 5: Agent synthesizes all results into a clear answer

Best Practices

PracticeWhy
Start with topics, then narrow Avoids wasting calls on repos where CodeWiki has limited coverage
Use structure before contents Lets the agent pick the right section instead of reading everything
Prefer section reads over search for broad topics Section content is deterministic and cached; search uses Gemini chat which is slower
Use search for specific implementation questions Gemini chat excels at pointed questions about code behavior
Handle errors gracefully Check status field; repos may not be indexed or responses can time out
Use owner/repo shorthand Cleaner and always works; full URLs are also accepted
Cache-aware timing First call may take 5-15s (Playwright render); subsequent calls within 5 min are cached

MCP Client Configuration

To connect your agent to CodeWiki MCP, add this server configuration to your MCP client:

VS Code (.vscode/mcp.json)

{
  "servers": {
    "codewikiMcp": {
      "type": "stdio",
      "command": "codewiki-mcp"
    }
  }
}

Claude Desktop

{
  "mcpServers": {
    "codewiki": {
      "command": "codewiki-mcp"
    }
  }
}

Custom Agent (Python)

# Example: connecting via subprocess
import subprocess, json

proc = subprocess.Popen(
    ["codewiki-mcp"],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    text=True
)
# Send MCP JSON-RPC requests to proc.stdin
# Read MCP JSON-RPC responses from proc.stdout

Custom Agent (SSE)

# Start server in SSE mode
codewiki-mcp --sse --port 8080

# Then connect your agent to http://localhost:8080/sse

Agent Error Handling

Teach your agent to handle CodeWiki MCP errors gracefully:

# In your agent's tool-call handler:

result = call_mcp_tool("search_code_wiki", {
    "repo_url": repo,
    "query": question
})

response = json.loads(result)

if response["status"] == "error":
    code = response.get("code", "UNKNOWN")

    if code == "TIMEOUT":
        # Retry once, or fall back to read_wiki_contents
        pass
    elif code == "NO_CONTENT":
        # Repo may not be indexed — inform the user
        pass
    elif code == "RATE_LIMIT":
        # Wait and retry, or switch to cached tools
        pass
    elif code == "VALIDATION":
        # Fix the input (bad repo_url format, etc.)
        pass
    else:
        # Log and inform user of unexpected error
        pass

Token Efficiency Guide

CodeWiki MCP tools vary dramatically in how many tokens they return. Understanding this helps you write efficient agents and save on LLM costs.

Tool Token Cost Comparison

ToolTypical TokensUse For
read_wiki_structure500–1,500Table of contents / discovery
list_code_wiki_topics2,000–4,000Titles + content previews
read_wiki_contents (section)500–5,000Targeted section content
read_wiki_contents (paginated)3,000–8,000Multiple sections at once
search_code_wiki1,000–6,000Specific questions

Best Token-Efficiency Workflow

  1. Start with read_wiki_structure (cheapest, ~800 tokens)
  2. Read only the sections you need with read_wiki_contents(section_title=...)
  3. Use pagination (offset/limit) to avoid loading the entire wiki
  4. Reserve search_code_wiki for questions that sections can't answer
